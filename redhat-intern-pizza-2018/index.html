What is OpenSCAP?

First some terminology

  • SCAP stands for Security Content Automation Protocol
  • OVAL stands for Open Vulnerability and Assessment Language
  • When we say security policy or benchmark, we really mean "a set of rules created by a long-winded standards body".
  • When we say tailoring, we really mean "choose a subset of the rules that we care about."

Okay, so really, what is OpenSCAP?

  • Three major projects: OpenSCAP, SCAP Security Guide, and SCAP Workbench
  • Several minor projects: Anaconda plugin, OpenSCAP Daemon
  • SCAP Security Guide builds security profiles from sets of rules.
  • OpenSCAP scans a local or remote system against a particular benchmark.
  • SCAP Workbench can tailor guides and provides a graphical scanning tool.

That's nice, but who uses OpenSCAP?

  • US Government:
    • Military for STIG compliance
    • DoD for C2S (Commercial Cloud Services) baseline
    • Affiliated Contractors: NIST 800-171
  • Financial Institutions: PCI-DSS profile
  • Medical Institutions: HIPAA profile

What is the goal this Summer?

Make SCAP Security Guide easier to contribute to.

What makes it difficult currently?

  • Structured for programmers, not subject-matter experts
  • Siloed projects: different (Linux) operating systems shared relatively little policy
  • Confusing project organization: data about a rule exists in multiple separate tree

Well, how should we fix it?

  • Easy to write rules, documentation, and fixes (YAML, markdown, structure, etc.)
  • Merge all Linux platforms into a shared linux_os guide
  • Reorganize rules such that there is a shared directory for all associated information
  • Link to upstream location for a rule in the HTML reports
